## Directory Structure

The repository already contains a `terraform/` directory with upstream GCP code.
To keep it untouched, all EKS infrastructure lives in a dedicated subdirectory:

```
terraform/
├── main.tf          ← upstream (untouched)
├── variables.tf     ← upstream (untouched)
├── memorystore.tf   ← upstream (untouched)
├── providers.tf     ← upstream (untouched)
├── output.tf        ← upstream (untouched)
├── terraform.tfvars ← upstream (untouched)
├── README.md        ← upstream (untouched)
└── eks/
    ├── terraform.tf
    ├── variables.tf
    ├── vpc.tf
    ├── eks.tf
    ├── bastion.tf
    ├── data.tf
    └── outputs.tf
```

`terraform init` and `terraform apply` are always run from `terraform/eks/`.

## Design Decisions

### Naming

All resource names are derived from `var.project_name` (`microservices-demo`). Nothing is hardcoded.

- VPC: `microservices-demo-vpc`
- EKS cluster: `microservices-demo-eks`
- Bastion: `microservices-demo-bastion`

### EKS — Kubernetes 1.35

Version pinned at `1.35` via `var.kubernetes_version` for reproducibility.

### EKS — Authentication Mode

`authentication_mode = "API_AND_CONFIG_MAP"` — both the Access Entry API and the legacy
`aws-auth` ConfigMap are active simultaneously. This is required for compatibility with
tools that still write to `aws-auth`.

`enable_cluster_creator_admin_permissions = true` — the IAM identity running
`terraform apply` gets cluster-admin automatically. No manual `aws-auth` patching needed.

### EKS — Private API Endpoint

```hcl
endpoint_public_access  = false
endpoint_private_access = true
```

The API server is not reachable from the internet. All `kubectl` and `helm` commands
must be run from the bastion host. A dedicated security group explicitly allows
port 443 from the bastion to the cluster.

### Node Group

- Instance type: `t3.medium`
- AMI: `AL2023_x86_64_STANDARD` (Amazon Linux 2023, default from EKS 1.30+)
- Min: 2 / Desired: 2 / Max: 6
- `node_repair_config.enabled = true` — EKS auto-replaces unhealthy nodes
- `max_unavailable_percentage = 33` — rolling updates replace at most 33% of nodes at once

### Core Add-ons

`vpc-cni` and `eks-pod-identity-agent` use `before_compute = true` — they are installed
before any worker nodes join, preventing a race condition where pods schedule before
the CNI is ready.

### Bastion — Ubuntu 26.04

The AMI is resolved dynamically at apply time via a `data "aws_ami"` block:

```hcl
filter {
  name   = "name"
  values = ["ubuntu/images/hvm-ssd-gp3/ubuntu-resolute-26.04-amd64-server-*"]
}
```

Owner: `099720109477` (Canonical). `most_recent = true` always picks the latest patch release.

Security hardening:
- IMDSv2 enforced (`http_tokens = required`) — prevents SSRF-based metadata credential theft
- EBS root volume encrypted at rest (`gp3`, 20 GB)
- SSH ingress locked to the operator's current public IP — resolved at apply time via `checkip.amazonaws.com`
- SSH key pair generated by Terraform, saved locally as `.pem` with `0400` permissions

### VPC

`enable_dns_hostnames = true` and `enable_dns_support = true` are required by EKS
for node DNS resolution. Without these, the cluster will not become operational.

Subnet tags for AWS Load Balancer Controller auto-discovery:

```hcl
# Public subnets
"kubernetes.io/role/elb" = "1"

# Private subnets
"kubernetes.io/role/internal-elb" = "1"
```

### Global Tags via `default_tags`

```hcl
default_tags {
  tags = {
    Project     = "microservices-demo"
    ManagedBy   = "terraform"
    Environment = var.environment
  }
}
```

All resources are tagged automatically — no per-resource tag blocks needed.

### S3 Backend

An S3 backend block is present in `terraform.tf` but commented out.
Uncomment and configure before the first `terraform apply` in a CI or team setting.
State is local until then.

## CI — Terraform Validate

A parallel job was added to `.github/workflows/terraform-validate-ci.yaml`.
The existing job was left untouched.

```yaml
# Validates the EKS infrastructure defined in terraform/eks/
eks-terraform-validate:
  runs-on: ubuntu-24.04
  steps:
    - uses: actions/checkout@v6
    - uses: hashicorp/setup-terraform@v4
    - name: terraform init & validate
      run: |
        cd terraform/eks/
        terraform init -backend=false
        terraform validate
```

- Both jobs run in parallel
- `terraform init -backend=false` — no AWS credentials needed in CI for validate
- Path trigger `terraform/**` already covers `terraform/eks/**`

## Issues Encountered

### 1. Security Group Description — Apostrophe Not Allowed

**Error:**
```
"ingress.0.description" doesn't comply with restrictions
("^[0-9A-Za-z_ .:/()#,@\[\]+=&;{}!$*-]*$"):
"SSH from operator's current public IP"
```

**Fix:** Removed the apostrophe from the ingress description in `bastion.tf`:
```
"SSH from operators current public IP"
```

AWS Security Group descriptions do not allow single quotes.

### 2. Deprecated Attribute — `data.aws_region.current.name`

**Warning:**
```
name is deprecated. Use region instead.
```

**Fix:** Replaced all occurrences of `data.aws_region.current.name` with
`data.aws_region.current.region` in `outputs.tf`.
